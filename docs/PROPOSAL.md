## 一、产品定义

### 1.1 一句话定义

让政企顾问与工程师在加密、合规、隔离的环境下,也能用上 AI 编码与研究助手——并把每个人的工作成果沉淀为组织资产。

### 1.2 不是什么

明确边界,避免越做越散:

- ❌ **不是**另一个 Cursor / Copilot —— 它们假设代码可自由读写,政企环境跑不起来
- ❌ **不是**另一个 Dify / Coze —— 它们是工作流编排,不管端侧合规和多用户
- ❌ **不是**破解加密系统 —— 这条线永远不碰
- ❌ **不是**通用 SaaS 产品 —— 私有化部署是核心模式而非妥协

### 1.3 是什么

- ✅ 在客户/公司内网部署的多租户 AI Agent 平台
- ✅ 让 AI 工具与企业合规体系(加密客户端、SSO、DLP、审计)深度集成
- ✅ 把顾问的隐性知识转化为组织级可复用 Skill 资产

### 1.4 三类目标用户

| 用户 | 场景 | 核心价值 |
|------|------|---------|
| 咨询顾问(主)| 客户现场分析代码、写报告、跨项目知识复用 | 提效 + 可携带的方法论 |
| 政企工程师 | 加密代码库内的 AI 辅助开发 | 不破坏合规前提下用 AI |
| 知识管理者 | 把组织 know-how 沉淀为可复用资产 | 把人天计费业务变成可规模化能力 |

## 二、市场机会

### 2.1 痛点验证

- 政企客户代码/数据加密 → 顾问没法用 AI 高效分析
- 顾问拿脱敏数据回公司 → 内部用 AI 又有合规边界
- 用了 AI 出方案 → 客户问"你们怎么保证我数据安全"
- 现有 SaaS AI 工具(Cursor、Copilot、ChatGPT)在政企环境**基本无法落地**

### 2.2 市场规模(粗估)

- 国内政企数字化咨询市场规模千亿级
- 其中"有 AI 提效需求但受合规约束"的细分市场,正是中大咨询及同类公司的客户基本盘
- 一旦产品化,可作为咨询交付的**标准化能力包**

### 3.2 核心数据流

**场景**:顾问 A 在企微问"客户 X 项目的支付模块有什么风险"

```
① 顾问 A 企微发消息
  ↓
② 统一 Gateway 接收 → 解析企微 user_id
  ↓
③ 认证授权层 → 映射到 AegisAgent user_id → 校验权限(能否访问客户 X)
  ↓
④ 进程编排器 → 检查顾问 A 的 profile 进程
   ├── 运行中 → 直接转发
   └── 未运行 → 冷启动 profile(~3s)→ 转发
  ↓
⑤ Hermes profile 进程接管
  ↓
⑥ 合规中间层拦截:
   ├── 授信代理:识别要读的文件密级
   ├── 数据分级:决定调用哪个模型(私有 LLM)
   ├── 脱敏引擎:去除客户名/IP/敏感字段
   └── 审计日志:记录"谁在何时读了什么"
  ↓
⑦ Hermes Agent 调用工具 + 模型
  ↓
⑧ 回写阶段:
   ├── 输出加密(同源加密策略)
   └── 知识沉淀:产出的方法论 → 组织 Skill 仓候选
  ↓
⑨ 回包给顾问 A 企微
```

## 四、多租户控制平面(核心)

### 4.1 为什么单独拎出来讲

这是整个产品**最难、最有价值、也最容易被低估**的部分。Hermes 自带 profile / board / tenant 三层隔离原语,但**它不是多用户 SaaS**——你需要把它"包装"成一个能给上百号顾问共享的服务。

### 4.2 Hermes 现有的隔离原语

| 层 | 机制 | 隔离强度 | 用途 |
|----|------|----------|------|
| Profile | `HERMES_HOME` 环境变量切换 | 强(进程级)| 不同用户 |
| Board | 任务队列 + workspace 目录隔离 | 强 | 不同项目 |
| Tenant | board 内的命名空间软过滤 | 弱 | 项目内的客户子模块 |

**关键事实**:Hermes 的代码库里有 119+ 个文件通过 `get_hermes_home()` 解析路径,所以 profile 切换是真正的全状态隔离(config、sessions、memory、skills、cron、gateway PID、logs 全部隔离)。

### 4.3 关键缺口

| 缺口 | 描述 | 影响 |
|------|------|------|
| 单用户产品假设 | Hermes 设计是"一个人在自己机器上跑" | 缺 SSO、配额、运维面板 |
| 进程开销 | 每个 profile 一个 Python 进程 | 100 用户 = 内存爆炸 |
| Gateway 绑死 profile | 默认每个 profile 一个 bot token | 100 用户 = 100 个机器人,不可能 |
| 无组织级共享 | Skill 是 profile 私有 | 知识无法沉淀到组织 |

### 4.4 八个子模块

#### 模块 A:认证授权 (Auth)

**目标**:把企业身份系统接入 Hermes,让用户身份贯穿整个调用链。

**设计**:
- 对接企业 SSO(LDAP / OAuth / 企微 / 钉钉开放接口)
- 维护映射表:`enterprise_user_id ↔ aegisagent_user_id ↔ hermes_profile_path`
- 首次登录自动创建 profile:`HERMES_HOME=/data/profiles/{aegisagent_user_id}`
- RBAC 权限模型:用户 × 角色 × 客户项目 三维度
  - 例:"高级顾问" + "客户 X 项目" 权限 = 可访问 `/projects/customerX/*`
- Token 有效期与企业 SSO 会话同步

**技术选型**:Keycloak / Casdoor 任选;自研也不复杂

---

#### 模块 B:统一 Gateway 路由 (Router)

**整个改造里最关键的一环。**

**问题**:Hermes 默认每个 profile 一个 gateway 进程 + 一个 bot token。100 个顾问 = 100 个企微机器人 = 不可能。

**方案**:**单 Gateway + 多 Profile 路由**

```
单一企微机器人 (AegisAgent bot)
    ↓
Gateway 层(自研,Python/Node)
    ├── 接收企微回调消息
    ├── 从 message.from_user 解析企业 user_id
    ├── 查询认证模块,得到对应 aegisagent_user_id
    ├── 通过 IPC / HTTP 转发给对应 profile 进程
    └── 收集 profile 输出,回写到企微
```

**实现要点**:
- **Fork hermes-agent 仓库做 patch**:Hermes 现有 gateway 模块绑死单 profile,需重构
- 各平台 gateway 适配:钉钉、飞书、企微 API 不同,封装统一抽象层
- 长连接 + Webhook 双模式支持
- 消息格式标准化:把企微的图文混排转换为 Hermes 能理解的格式

---

#### 模块 C:进程编排 (Orchestrator)

**问题**:100 个 profile 同时常驻 = 内存爆炸。

**方案**:Serverless 化的 Hermes 进程池

**生命周期状态机**:

```
状态机:
  cold(磁盘上的 profile 数据,无进程)
    ↓ [收到消息触发]
  warming(冷启动中,~3s)
    ↓
  active(运行中,处理消息)
    ↓ [闲置 N 分钟]
  idle(待回收)
    ↓ [graceful shutdown]
  cold(回到起点,数据持久化)
```

**关键参数**:
| 参数 | 默认值 | 调优方向 |
|------|--------|----------|
| 闲置回收时间 | 15 分钟 | 高频用户更长 |
| 最大并发 active | 30 | 看服务器规格 |
| 冷启动预算 | 3 秒 | 通过 SnapStart 类似优化压到 1 秒内 |
| 单 profile 内存上限 | 4GB | cgroup 限制 |
| 单 profile CPU 上限 | 2 vCPU | cgroup 限制 |

**预热策略**:
- 工作时间(9:00-18:00)预拉起 TOP 20 活跃用户的 profile
- 用户首字符出现时显示"思考中"占位,掩盖冷启动

**崩溃恢复**:
- profile 进程挂了自动重启
- 不丢数据(Hermes 本身 SQLite 持久化)

**技术选型**:
- MVP 阶段:**PM2 + 自研调度脚本**
- 中量级:**Nomad** 或 **Docker Compose + 自研编排**
- 规模化:**Kubernetes + Operator**

#### 模块 D:配额与计费 (Quota)

**为什么需要**:防止个别用户(或失控脚本)把资源吃光,同时为按部门/项目分摊成本做准备。

**配额维度**:
- Token 消耗(月度上限 / 日上限)
- 模型调用次数
- 同时活跃 profile 数(每用户最多 1 个)
- 存储配额(profile 数据 + skills + memory)
- 工具调用次数(尤其外部 API)

**计费场景**(对客户交付时):
- 按部门维度统计成本,月度账单
- 不同密级数据走不同模型 → 不同费率
- 异常告警:"市场部本月 AI 消耗已达上限的 80%"

---

#### 模块 E:审计日志 (Audit)

**这是政企客户的硬需求,不是 nice-to-have。**

**记录内容**:
- 每次会话的开始/结束时间、用户身份
- 每次工具调用(读了哪个文件、调了哪个外部 API)
- 每次模型调用(prompt + response 的 hash,可选完整内容)
- 每次外发(脱敏前后对比、审批人、审批时间)
- 每次配置变更

**存储要求**:
- 日志独立存储,**不可篡改**(WORM 策略,可选区块链锚定)
- 保留期至少 6 个月(等保要求)
- 审计员账号专属界面查询,普通用户看不到他人日志

**关键设计**:**审计日志本身是产品价值的一部分**——客户做合规检查时能直接调出报表,这是中大对客交付的核心卖点之一。

---

#### 模块 F:组织级 Skill 仓 (Skill Vault)

**AegisAgent 的"杀手锏",一定要做好。**

**问题**:Hermes 的 skill 默认是 profile 私有的。100 个顾问各自的 skill 没法共享 = 组织无法积累知识。

**方案**:三级 Skill 体系

```
个人 Skill(profile 私有)
    ↓ 顾问主动提交
项目 Skill(客户项目内可见)
    ↓ 项目结束 + 脱敏审核
组织 Skill(全公司可用)
    ↓ 沉淀为标准方法论
行业 Skill 模板(对外交付物的一部分)
```

**审核流程**:
1. 顾问产出有价值的 skill(例:"交通行业 ETC 数据分析方法")
2. 提交到组织 Skill 仓 → 触发脱敏审核
3. 自动检查:是否含客户名、内网 IP、敏感字段
4. 人工审核(项目负责人 + 信息安全双签)
5. 通过 → 进入组织 Skill 仓 → 全公司顾问可用
6. 调用统计:看哪些 skill 被高频使用 → 反向激励顾问贡献

**商业价值**:
- 每个项目都在为组织积累资产,而不是项目结束清零
- 资产化的 skill 可作为对客交付物的一部分
- 顾问的"贡献度"成为绩效新维度
- 长期看,**组织 Skill 仓的丰富度 = 中大咨询的核心壁垒**

---

#### 模块 G:客户/项目隔离 (Tenant Isolation)

**问题**:同一顾问可能同时做 3 个客户的项目,客户 A 的数据和上下文绝对不能进客户 B 的对话。

**方案**:利用 Hermes 的 board 机制 + 自研项目切换

- 每个客户项目对应一个 Hermes board
- 顾问切换项目 = 切换 board(企微发 `/switch projectX` 或 Web UI 选择)
- 切换后,memory、history、skills 全部限定在该项目范围内
- **跨项目调用要显式确认**:"我能不能用客户 A 项目里学到的方法,帮你处理客户 B 项目?" → 默认拒绝,经审批后允许

**项目结束流程**:
- 客户项目结束 → board 归档
- 数据保留 X 天后,要么交付给客户、要么销毁
- 关键 skill 走脱敏审核,可能晋升到组织 Skill 仓

---

#### 模块 H:监控告警 (Monitoring)

**该看什么**:
- **系统健康**:profile 进程崩溃率、冷启动延迟、消息处理延迟 P99
- **业务指标**:DAU/WAU、消息量、Token 消耗、Skill 使用 Top N
- **异常行为**:
  - 某用户突然大量外发申请
  - 某 profile 持续高内存
  - 某客户项目数据访问异常

**技术选型**:Prometheus + Grafana + Loki(日志) + AlertManager

## 五、合规中间层

| 模块 | 核心功能 | 实现要点 |
|------|----------|----------|
| 授信代理 | Hermes 进程纳入加密客户端白名单 | 与亿赛通/深信服等做适配;识别文件密级元数据 |
| 数据分级路由 | 按密级路由到不同模型 | 公开 → 公网,内部 → 私有 LLM,机密 → 本地推理 |
| 脱敏引擎 | 出站前自动脱敏 | NER 命名实体识别(人名/客户名/IP)+ 规则替换 |
| 外发审批 | 一键申请 + OA 对接 | 状态机驱动:申请 → 审批 → 执行 → 留痕 |
| 输出加密回写 | AI 生成内容自动加密 | 调用加密客户端 API,匹配源文件密级 |

## 六、核心场景

### 场景 1:客户现场代码 Review

> 顾问在客户机房,通过企微向 AegisAgent 发问:"看下这个支付模块的架构合理吗"
>
> → Hermes 在客户内网读加密代码 → 私有 LLM 分析 → 企微回复结论
>
> → **全程数据不出客户内网**

### 场景 2:跨项目知识沉淀

> 顾问 A 在交通项目里教 Hermes "业务术语库"
>
> → 项目结束后,知识沉淀为 Skill,经脱敏审核进入中大公共知识库
>
> → 顾问 B 接下一个交通项目时,Hermes 已具备行业术语理解能力

### 场景 3:对客交付

> 中大向客户交付"合规 AI 工作台"作为咨询项目的一部分
>
> → 客户得到的不只是 PPT 报告,还有一个能持续运行的 AI 助手
>
> → 中大获得长期运维订单

## 七、技术栈选型

### 7.1 核心技术栈

| 层 | 选型 | 理由 |
|----|------|------|
| Hermes 底座 | Fork hermes-agent + Patch | MIT 协议可商用,功能匹配度最高 |
| 控制平面后端 | Python (FastAPI) + Go(性能敏感)| 与 Hermes 生态一致,FastAPI 团队熟 |
| 前端 | React + TypeScript | 团队成熟栈 |
| 进程编排 | MVP 阶段:PM2 + 自研脚本 / 后期:K8s | 渐进式 |
| 数据库 | PostgreSQL(用户/审计) + SQLite(profile)| 不动 Hermes 默认 |
| 消息队列 | Redis Stream(轻) / NATS(重) | 看规模 |
| 监控 | Prometheus + Grafana + Loki | 业界标准 |
| 部署 | Docker Compose(私有化) + K8s(规模化)| 私有化部署是核心场景 |

### 7.2 模型层

- **私有化 LLM**:Qwen3-32B / GLM-4 / DeepSeek-V3,信创认证版优先
- **公网 API**:Claude / GPT(仅在合规允许场景)
- **本地小模型**:用于脱敏 NER、敏感词检测(不要每次都打大模型)

### 7.3 集成方

- **加密系统**:亿赛通 CDG / 深信服 EDR / 绿盾 / IP-guard(优先适配 TOP3)
- **IM**:企微 / 钉钉 / 飞书
- **OA**:泛微 / 致远 / 蓝凌(对接外发审批流程)
- **IDE**:VSCode / JetBrains(Hermes 已有 ACP 支持)

## 八、商业化路径

### 阶段一:内部工具(0-3 个月)

- 在中大内部上线 AegisAgent 1.0
- 服务 1-2 个标杆项目,验证价值
- **度量指标**:顾问效率提升 %、项目交付周期缩短 %

### 阶段二:对客交付能力(3-9 个月)

- 包装成"合规 AI 转型咨询"产品
- 作为现有咨询项目的**增值交付物**销售
- **定价模式**:咨询费 + 部署实施费 + 年度运维费

### 阶段三:产品化输出(9-18 个月)

- 沉淀为标准化产品 + 行业模板(交通版 / 金融版 / 政务版)
- 探索独立 SKU 销售或合作伙伴渠道
- 与等保测评、信创生态打通,形成壁垒

## 九、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| Hermes 上游版本快速迭代,patch 一直要 rebase | 高 | 中 | 锁定主要版本,patch 模块化,贡献部分能力到上游 |
| 加密厂商不配合 | 中 | 高 | 优先选有 SDK / 开放 API 的厂商;白盒方案做 fallback |
| 政企客户对开源工具有顾虑 | 中 | 中 | 提供完整代码审计 + 第三方安全评估报告 |
| 中大不感兴趣 | 中 | 低 | 项目本身有独立价值,可作开源 / 求职作品 |
| 技术栈复杂度超出独立完成能力 | 中 | 中 | 严格控制 MVP 范围,先打通核心链路 |
| 多用户场景下 Hermes 出现未知 bug | 高 | 中 | 充分压测;关键路径有降级方案 |

## 十、为什么是基于 Hermes 而非 OpenClaw

| 维度 | Hermes Agent | OpenClaw | Cursor/Copilot |
|------|--------------|----------|----------------|
| 部署模式 | 自托管 ✅ | 自托管 ✅ | SaaS ❌ |
| 数据留存 | 全本地 ✅ | 全本地 ✅ | 云端 ❌ |
| 多端触达 | 18+ 平台(含钉钉/飞书/企微)✅ | CLI/IDE | IDE only |
| 持久记忆 | 跨会话学习循环 ✅ | 文件式记忆 | 会话内 |
| 协议 | MIT,可商用 ✅ | MIT ✅ | 闭源 |
| 适配场景 | 长期项目跟进 | 编码任务 | 编码任务 |

**结论**:Hermes 的"自托管 + 多端 + 持久记忆 + profile 隔离原语"四大特性,与政企咨询场景天然契合。OpenClaw 更偏编码工具,与"咨询顾问的 AI 助理"定位有偏差。

## 附录:命名说明

项目正式名称定为 **AegisAgent**。

**词源与寓意**:

- **Aegis(神盾)**——希腊神话中宙斯与雅典娜所持的神盾,象征**护佑与防御**。这与产品定位高度契合:在政企合规约束下,为 AI 工具提供一道"既能用、又安全"的护盾。
- **Agent**——明确产品形态,与底座 Hermes Agent 形成语义呼应,但又有独立的品牌识别度。

**为什么不沿用 HermesGov 等候选**:

| 候选 | 否决原因 |
|------|---------|
| HermesGov | 带 "Gov" 字眼可能让金融、能源、制造等非政府客户产生距离感 |
| Caduceus(双蛇杖)| 拼写复杂,中文市场难传播 |
| 政启 / 信驰 / 合规鹰 | 中文品牌国际化困难,GitHub 开源传播受限 |

**AegisAgent 的优势**:
- ✅ 语义与产品价值(安全护盾)直接对应
- ✅ 同时适配政企、金融、能源、咨询等多个目标行业
- ✅ 中英文双语市场都能传播(中文可音译为"神盾"或保留英文)
- ✅ GitHub 仓库地址简洁:`github.com/lewisoepwqi/AegisAgent`

**中文场景建议**:
- 正式文档:统一使用 **AegisAgent**
- 营销/对客场景可选副标题:"AegisAgent · 神盾智能体" 或 "AegisAgent · 政企合规 AI 工作台"
